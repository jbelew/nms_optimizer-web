# INP Bottleneck Audit — 2026-05-05

Audit of the NMS Optimizer web app focused on **Interaction to Next Paint (INP)** Core Web Vital bottlenecks. Scope: hot interaction paths in the grid (tap/double-tap), tech tree sidebar, and module selection dialog.

## Findings

Severity reflects realistic INP impact (a 200ms+ blocking task during a click is critical/high; a 50ms one is medium; minor optimizations are low).

### 1. Derived selectors run O(N) traversals on every subscription update — **high**

**File:** [`src/store/grid/gridStore.ts`](../src/store/grid/gridStore.ts) (L734–L794)

`hasTechInGrid`, `isGridFull`, `selectTotalSuperchargedCells`, `selectHasModulesInGrid`, `selectFirstInactiveRowIndex`, and `selectLastActiveRowIndex` are exposed as **getter functions on state**. Components subscribing via these selectors re-execute nested `flat().filter().every()` traversals on **every** grid mutation.

**Impact:** Because the 10×6 grid mutates on every cell tap/drag, every subscribed component (dozens of `TechTreeRow`s, `GridCell`s) reruns a full grid scan. This compounds with finding #2.

**Fix:** Extract these into precomputed primitive state updated inside actions (`setGrid`, `toggleCellActive`, `toggleCellSupercharged`, etc.):

```ts
// Add to state:
activeTechs: Set<string>;
isGridFull: boolean;
totalSuperchargedCells: number;

// Update inside set((state) => { ... }) for each cell mutation.
```

Then components select primitives: `useGridStore(s => s.activeTechs.has(tech))`.

### 2. `TechTreeRow` runs full-grid `hasTechInGrid` scans ~30× per cell tap — **high**

**File:** [`src/components/TechTreeRow/useTechTreeRow.ts`](../src/components/TechTreeRow/useTechTreeRow.ts) (L68)

```ts
const hasTechInGrid = useGridStore((state) => state.hasTechInGrid(tech));
```

This hook is mounted for **every tech category** (~30 instances). Each invocation re-iterates the entire 10×6 grid on every store update. Result: O(rows × cells) work per single cell tap.

**Fix:** After implementing #1, replace with a primitive selector against the precomputed `activeTechs` set:

```ts
const hasTechInGrid = useGridStore((state) => state.activeTechs.has(tech));
```

### 3. `TechTreeRow` subscribes to entire `techGroups` object without `useShallow` — **high**

**File:** [`src/components/TechTreeRow/useTechTreeRow.ts`](../src/components/TechTreeRow/useTechTreeRow.ts) (L69–L70)

```ts
const techGroups = useTechStore((state) => state.techGroups);
```

Returns the full object reference. Any mutation to any tech in `techGroups` triggers a re-render of **every** `TechTreeRow`.

**Fix:**

```ts
import { useShallow } from "zustand/react/shallow";

const techGroup = useTechStore(useShallow((state) => state.techGroups[tech]));
const modules = techGroup?.[0]?.modules || EMPTY_MODULES_ARRAY;
const moduleCount = techGroup?.[0]?.module_count || 0;
```

### 4. `TechTreeContent` recomputes & sorts entire tech tree on every render — **high**

**File:** [`src/components/TechTree/TechTreeContent.tsx`](../src/components/TechTree/TechTreeContent.tsx) (L63–L98)

`Object.entries(techTree).forEach(...).filter().map().sort(localeCompare)` runs on every render. Because the component subscribes to `isGridFull` (L61), every flip re-sorts the whole tree synchronously on the main thread.

**Fix:** Memoize the derivation:

```ts
const processedTechTree = React.useMemo(() => {
  const result: { [key: string]: TechTreeItem[] } = {};
  Object.entries(techTree).forEach(([category, technologies]) => {
    if (category === "recommended_builds" || category === "grid" || category === "grid_definition") return;
    const safeTechnologies = Array.isArray(technologies) ? technologies : [];
    result[category] = safeTechnologies
      .filter((tech): tech is TechTreeItem => typeof tech === "object" && tech !== null && "key" in tech)
      .map((tech) => ({ ...tech, modules: tech.modules || [], image: tech.image || null }))
      .sort((a, b) => a.label.localeCompare(b.label));
  });
  return result;
}, [techTree]);
```

### 5. `useTechModuleManagement` rebuilds module groups on every render — **high**

**File:** [`src/components/TechTreeRow/useTechModuleManagement.ts`](../src/components/TechTreeRow/useTechModuleManagement.ts) (L63–L94)

`coreModuleIds`, `nonCoreModuleIds`, and the `groupedModules` `forEach` are reconstructed on every render across all ~30 rows.

**Fix:** Wrap derivations in `useMemo`:

```ts
const { coreModuleIds, nonCoreModuleIds, groupedModules } = React.useMemo(() => {
  const coreModuleIds = modules.filter((m) => m.type === "core").map((m) => m.id);
  const nonCoreModuleIds = modules.filter((m) => m.type !== "core").map((m) => m.id);
  const groups: { [key: string]: typeof modules } = {
    core: [], bonus: [], upgrade: [], reactor: [],
    cosmetic: [], atlantid: [], trails: [], figurines: [],
  };
  modules.forEach((module) => {
    if (module.label?.toLowerCase().includes("figurine")) {
      groups.figurines.push(module);
      return;
    }
    const type = module.type || "upgrade";
    if (groups[type]) groups[type].push(module);
    else groups.upgrade.push(module);
  });
  return { coreModuleIds, nonCoreModuleIds, groupedModules: groups };
}, [modules]);
```

### 6. `GridCell` is not wrapped in `React.memo` — **high**

**File:** [`src/components/GridCell/GridCell.tsx`](../src/components/GridCell/GridCell.tsx) (L157–L245)

The component uses `useShallow` internally for cell state, but the component itself isn't memoized. Any re-render of the parent `GridTable` (e.g., `solving` flip, `sharedGrid` change, parent transition) re-renders **all 60 cells**.

**Fix:**

```ts
export default React.memo(GridCell);
```

Props are primitives (`rowIndex`, `columnIndex`, `isSharedGrid`), so the default shallow compare suffices.

### 7. `sendEvent` (GA4) runs synchronously inside `startTransition` — **medium**

**File:** [`src/components/TechTreeRow/TechTreeRow.tsx`](../src/components/TechTreeRow/TechTreeRow.tsx) (L443–L453)

```ts
startTransition(() => {
  setIsOpen(open);
  sendEvent({ category: "engagement", ... });
});
```

`startTransition` runs its callback synchronously before yielding. Including the analytics payload inside delays the dialog-open paint.

**Fix:** Move analytics outside the transition and defer:

```ts
startTransition(() => setIsOpen(open));

const fireAnalytics = () => sendEvent({ /* ... */ });
if ("requestIdleCallback" in window) {
  window.requestIdleCallback(fireAnalytics, { timeout: 1000 });
} else {
  setTimeout(fireAnalytics, 0);
}
```

### 8. Inline `onClose` defeats memoization on heavy `ModuleSelectionDialog` — **medium**

**File:** [`src/components/TechTreeRow/TechTreeRow.tsx`](../src/components/TechTreeRow/TechTreeRow.tsx) (L500)

```tsx
<ModuleSelectionDialog
  ...
  onClose={() => handleOpenChange(false)}
/>
```

The unstable inline function defeats any memo wrapping the large dialog/checkbox tree. Every parent re-render reconstructs the closure and re-renders the dialog subtree.

**Fix:**

```ts
const handleClose = React.useCallback(() => handleOpenChange(false), [handleOpenChange]);
// ...
<ModuleSelectionDialog ... onClose={handleClose} />
```

(Combine with wrapping `ModuleSelectionDialog` itself in `React.memo` if it isn't already.)

## Recommended Application Order

To maximize INP improvement with minimal risk:

1. **#6** — Single-line `React.memo` change. Immediate cell-tap win.
2. **#1** + **#2** — Together; precomputed primitive state plus the simplified selector. Largest single win for grid interactions.
3. **#3** — `useShallow` selector swap, complements #2.
4. **#5** — `useMemo` around module grouping.
5. **#4** — `useMemo` around tech tree sort.
6. **#7** — Defer analytics out of `startTransition`.
7. **#8** — `useCallback` for dialog `onClose`.

## Out of Scope (No INP Issues Found)

- `src/main.tsx` bootstrap correctly defers Sentry/GA4/SW registration via `requestIdleCallback` after `app-ready`.
- `useGridDeserializer` runs only on URL load, not interaction.
- `useOptimize` already wraps progress updates in `startTransition` on the WebSocket path.
- `GridTable` already uses `useDeferredValue` and `React.useMemo` for its row map.
